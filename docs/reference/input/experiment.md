# Experiment input format

An experiment is described by a single JSON file. Its top-level object is the [ExperimentSpec](#experimentspec); every axis listed there forms a cartesian product of scenarios. Topology files are referenced by path and documented separately.

## ExperimentSpec { #experimentspec }

Specification of an experiments. The experiment defines what should be simulated and how. The experiment file is the root of the simulation, and thus the most important file for your simulation. Every set-valued property below is one axis of a cartesian product: one scenario is run for each combination of the axes, and each scenario is repeated `runs` times with successive seeds.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | String | no | Human-readable name of the experiment. When left empty a unique name is generated automatically. Used as a prefix for output folders. |
| `outputFolder` | String | no | Directory that raw output is written to, under `<outputFolder>/raw-output/`. Default output folder is "output". |
| `initialSeed` | Int | no | Seed used to initialize the random generators. Successive runs of a scenario increment from this value. Default value is 0. |
| `runs` | Int | no | Number of runs (seeds) executed per scenario. Must be greater than 0. Default value is 1. |
| `topologies` | Set&lt;[TopologyPathSpec](#topologypathspec)&gt; | **yes** | Datacenter topologies to simulate, each referenced by a path to a topology JSON file. |
| `workloads` | Set&lt;[WorkloadSpec](workload.md#workloadspec)&gt; | **yes** | Workload traces to replay against the topologies. |
| `allocationPolicies` | Set&lt;[AllocationPolicySpec](allocation-scheduling.md#allocationpolicyspec)&gt; | no | Scheduling policies that decide how and when to schedule tasks. |
| `failureModels` | Set&lt;[FailureModelSpec](failures.md#failuremodelspec)?&gt; | no | Failure models decide when failures are injected during the simulation, their intensity, and duration. Failures cause hosts to crash, interrupting all running tasks, and making the hosts unavailable for a period. Default no failures are injected. |
| `maxNumFailures` | Set&lt;Int&gt; | no | Number of failures after which a task is considered permanently failed. Default is 10. |
| `checkpointModels` | Set&lt;[CheckpointModelSpec](checkpointing.md#checkpointmodelspec)?&gt; | no | Checkpointing behavior used to recover work after a failure. Default no checkpointing is used. |
| `exportModels` | Set&lt;[ExportModelSpec](export.md#exportmodelspec)&gt; | no | Telemetry export settings controlling which output files and columns are written. See `ExportModelSpec` for the Default exportModel. |

## taskstopper { #taskstopper }

Specification of a task stopper, which prematurely stops tasks when the forecasted carbon intensity stays above a threshold.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `windowSize` | Int | no | Number of past samples used to build the forecast. Must be positive. |
| `forecast` | Boolean | no | Whether a forecast is used instead of the current carbon intensity. |
| `forecastThreshold` | Double | no | Carbon-intensity threshold, as a fraction between 0.0 and 1.0, above which tasks are stopped. |
| `forecastSize` | Int | no | Number of samples that are forecasted into the future. Must be positive. |
